# SSL Setup Wizard Design

**Date:** 2026-03-27  
**Status:** Approved

## Overview

Add a web-based setup wizard that allows users to configure HTTPS access during initial deployment. Users choose between:
1. **Domain with Let's Encrypt** - Trusted SSL certificates for users with a domain name
2. **IP-only with self-signed cert** - HTTPS via self-signed certificate for users without a domain

## Requirements

- Initial setup only (one-time configuration, not runtime switchable)
- Web-based wizard (accessed over HTTP initially, upgrades to HTTPS after setup)
- Caddy bundled in docker-compose stack for TLS termination
- Let's Encrypt for domain-based SSL (auto-provisioned by Caddy)
- Self-signed certificates for IP-only access

## User Flow

```
Fresh Deploy:
1. User runs: docker compose up -d
2. Caddy starts in HTTP mode (port 80)
3. User accesses http://<vps-ip>
4. App detects first run → redirects to /setup
5. Setup wizard presents options:
   - "I have a domain name" → enter domain → Let's Encrypt
   - "IP address only" → self-signed certificate
6. Backend configures Caddy and restarts it
7. User redirected to HTTPS URL
```

## Architecture

```
┌─────────────────────────────────────────────────────────────────┐
│                        Docker Compose Stack                      │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│   ┌─────────────┐      ┌─────────────┐      ┌─────────────┐     │
│   │   Caddy     │ :443 │   Web       │ :3000│   API       │:8000│
│   │  (reverse   │─────►│  (frontend) │      │  (backend)  │     │
│   │   proxy)    │ :80  │             │      │             │     │
│   └─────────────┘      └─────────────┘      └─────────────┘     │
│         │                                          │             │
│         │              ┌─────────────┐             │             │
│         └─────────────►│  Caddyfile  │◄────────────┘             │
│           reads        │  (volume)   │   writes                  │
│                        └─────────────┘                           │
│                                                                  │
│   Shared volume: ./caddy/Caddyfile                              │
│   Cert storage:  ./caddy/data (Let's Encrypt certs)             │
│                  ./caddy/certs (self-signed certs)              │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

- Caddy is the only container exposed to the internet (ports 80/443)
- API writes Caddyfile to shared volume, then signals Caddy to reload
- Self-signed certs generated by API, stored in mounted volume
- Let's Encrypt certs managed entirely by Caddy

## Backend Components

### New API Endpoints

| Endpoint                | Method | Purpose                            |
| ----------------------- | ------ | ---------------------------------- |
| `/api/v1/setup/status`    | GET    | Check if initial setup is complete |
| `/api/v1/setup/configure` | POST   | Submit access mode configuration   |

### Request Schema

```json
{
  "mode": "domain" | "ip_only",
  "domain": "example.com"  // required if mode=domain
}
```

### New Services

| Component           | Responsibility                               |
| ------------------- | -------------------------------------------- |
| `SetupService`      | Orchestrates setup flow, writes config       |
| `CaddyConfigWriter` | Generates Caddyfile from templates           |
| `CertGenerator`     | Creates self-signed certs (for IP-only mode) |

### Database

New `SetupStatus` model to persist setup state.

### Caddyfile Templates

**Domain mode (Let's Encrypt):**
```
{domain} {
    reverse_proxy web:3000
    handle /api/* {
        reverse_proxy api:8000
    }
}
```

**IP-only mode (self-signed):**
```
:443 {
    tls /certs/cert.pem /certs/key.pem
    reverse_proxy web:3000
    handle /api/* {
        reverse_proxy api:8000
    }
}
```

## Frontend Components

### New Routes

| Route    | Purpose                               | Auth required |
| -------- | ------------------------------------- | ------------- |
| `/setup` | Initial setup wizard (first run only) | No            |

### New Components

| Component            | Purpose                                   |
| -------------------- | ----------------------------------------- |
| `SetupWizard`        | Main wizard container                     |
| `AccessModeSelector` | Radio buttons for domain vs IP-only       |
| `DomainInput`        | Domain name input with validation         |
| `SetupComplete`      | Success screen with redirect/instructions |

### Frontend Flow

1. App checks `/api/v1/setup/status` on load
2. If `setup_complete: false` → redirect to `/setup`
3. Wizard submits to `/api/v1/setup/configure`
4. On success → show completion message → redirect to HTTPS URL

## Error Handling

| Error Case                         | Handling                                                 |
| ---------------------------------- | -------------------------------------------------------- |
| Domain DNS not pointing to VPS     | Let's Encrypt will fail → show error with instructions   |
| Ports 80/443 blocked by firewall   | Let's Encrypt will fail → show error about opening ports |
| Caddy fails to restart             | API catches error → show error, keep HTTP mode           |
| Self-signed cert generation fails  | Show error, allow retry                                  |
| User accesses HTTP after HTTPS setup | Caddy auto-redirects to HTTPS                          |

### Caddy Restart Mechanism

1. API writes new Caddyfile
2. API calls `docker exec caddy caddy reload --config /etc/caddy/Caddyfile`
3. If reload fails, API keeps old config and reports error to user
